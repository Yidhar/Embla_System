# 09-工具调用与任务执行规范（Naga）

> 目的：统一“任务发布、计划审阅、执行闭环”的工具使用方式，避免口头承诺与实际执行脱节。

## 1. 总原则

1. 默认执行模式为：**任务发布 + 计划审阅 + 结果验收**。
2. 代码实现/修复任务优先通过 **Codex（codex-cli / ask-codex）** 下发与执行。
3. 本地 native 工具用于：
   - 读取上下文（文档、代码、git状态）
   - 轻量验证（构建、编译、检索）
   - 结果审阅与证据收集
4. 非必要不手工改仓库；如需手工落盘（例如文档规范、紧急热修）必须在回执中显式说明。

---

## 2. 工具选择策略

### 2.1 Codex 主链路（开发任务）

适用：
- 新功能开发
- 缺陷修复
- 重构与批量改造

执行：
- 使用 `codex-cli.ask-codex` 下发任务，任务单必须包含：
  - 目标
  - 变更文件范围
  - 验收命令
  - 约束（最小改动/兼容性/不触碰范围）

### 2.2 Native 辅链路（审阅与验证）

适用：
- 读取文档/文件
- git diff 与状态核验
- 构建与语法验证

常用：
- `list_files` / `read_file` / `search_keyword`
- `git_status` / `git_diff` / `git_log`
- `run_cmd`（npm build、py_compile 等）

---

## 3. 标准执行流程（闭环）

1. **定位与读取**：确认需求、入口文件、影响范围。
2. **发布任务**：用 Codex 下发明确任务单。
3. **审阅变更**：查看修改文件与 diff，确认是否符合任务边界。
4. **执行验证**：运行最小必要验证命令。
5. **结论回传**：输出“已完成项 + 证据 + 风险项 + 下一步”。

---

## 4. 失败与降级策略

### 4.1 Codex 超时/离线

若出现以下错误：
- `codex-cli appears offline`
- `ask-codex timed out`

处理顺序：
1. 记录失败原因与时间。
2. 尝试重试（可提高 timeout）。
3. 若仍失败，进入降级方案：
   - 继续用 native 完成“可做的审阅与验证”
   - 需要写入时由用户确认是否启用手工落盘
4. 回执中必须注明“本次是否由 Codex 执行”。

### 4.2 参数错误

如 `prompt required`：
1. 先修复参数再重试。
2. 不得在参数错误状态下宣称“任务已下发成功”。

## 6. 本项目约定（前后端分离场景）

1. 前端 API 基址统一支持环境变量（如 `VITE_API_BASE_URL`）。
2. 避免前端直接端口直连后端子服务，优先经 BFF（apiserver）聚合。
3. TTS/Agent 等能力统一走 BFF 路由暴露。
4. 验收至少包含：
   - 前端构建（`npm run build`）
   - 后端语法检查（`python -m py_compile ...`）
