# 03 C++/Qt UI 改造工程评估

## 1. 评估结论（先给结论）

结论：可改造，但建议采用“保留 Python 后端 + 重写桌面前端为 Qt”路线，不建议一阶段做全栈 C++ 重写。

推荐路线：

1. 保留现有后端服务（`apiserver`/`agentserver`/`mcpserver`/`summer_memory`）。
2. 以 Qt（优先 Qt Quick/QML）重写 Electron + Vue 前端。
3. 通过本地 HTTP/SSE 协议与后端对接，先做功能对齐，再做体验优化。

不推荐路线：

1. 同时替换 UI 与后端为 C++，风险与周期都会显著上升。
2. 在缺乏明确性能瓶颈证据前，直接做跨语言全栈重写。

## 2. 当前前端能力基线（需要对齐的功能）

当前 Electron 前端不只是静态 UI，包含较重的桌面壳层能力：

1. 无边框窗口控制（最小化、最大化、关闭、拖动区）。
2. 悬浮球模式（`classic/ball/compact/full` 四态、窗口动画、位置管理）。
3. 托盘、全局快捷键、开机自启。
4. 屏幕捕获源枚举与截图回传。
5. 自动更新（`electron-updater`）。
6. 启动后端子进程、解析后端进度与错误。
7. Vue 路由多页面（聊天、模型、记忆、意识海、技能工坊、配置）。
8. Live2D 相关渲染与交互（前端依赖 `pixi-live2d-display`、`pixi.js`）。

说明：Qt 改造不是纯 UI 重画，而是“桌面平台能力 + 交互系统 + 渲染能力”整体替换。

## 3. 可选改造方案对比

### 方案 A：仅替换 UI（推荐）

定义：

- 后端全保留（Python）。
- 前端由 Electron+Vue 改为 Qt（QML/C++）。
- 通信继续使用现有 API（HTTP/SSE）。

优点：

- 业务逻辑复用最多，风险可控。
- 可分阶段上线，支持灰度迁移。
- 与现有接口兼容，回滚成本低。

难点：

- 需要在 Qt 里实现 SSE 流解析、悬浮球行为、截图、更新、托盘等系统能力。
- Live2D 渲染链需重新落地（可选原生 SDK 或暂时降级）。

### 方案 B：UI + 部分本地能力改写

定义：

- 方案 A 基础上，将部分前端侧的 Node/Electron 本地能力迁移到 C++ 服务层。

优点：

- 桌面能力统一在 C++，可减少桥接层。

难点：

- 跨进程架构设计更复杂，调试成本增高。

### 方案 C：全栈 C++ 重写（不建议一期）

定义：

- Python 后端能力迁到 C++（包括 LLM 代理、任务调度、记忆图谱、工具路由等）。

问题：

- 当前后端接口规模大（API 与 Agent 路由较多）。
- 第三方生态迁移成本高（FastAPI/async 生态、AgentServer 集成、现有脚本化能力）。
- 工期长且高风险，业务迭代会明显停滞。

## 4. Qt 技术落地建议

建议基础栈：

1. UI：Qt 6 + Qt Quick/QML（复杂动画与状态机更自然）。
2. 网络：`QNetworkAccessManager` + 自定义 SSE 解析器。
3. 异步：`QObject + signals/slots` + 线程池。
4. 系统能力：
   - 托盘：`QSystemTrayIcon`
   - 开机自启：平台专用封装（Windows 注册表/任务计划）
   - 截图：`Qt Multimedia` 或平台 API 封装
5. 打包：`windeployqt` + 安装器（Qt IFW / NSIS）。
6. 自动更新：可选 `WinSparkle` 或自建增量更新逻辑。

Live2D 建议：

1. 若需强一致复刻，优先接入 Live2D Cubism Native SDK。
2. 若短期以功能上线为主，可分阶段：
   - 阶段 1：无 Live2D 或静态占位
   - 阶段 2：补齐 Live2D 与表情/追踪联动

## 5. 风险评估

### 高风险

1. 悬浮球四态与窗口动画行为复刻（平台差异明显）。
2. Live2D 渲染链迁移（图形后端、性能、资源格式适配）。
3. 自动更新链路替换（发布与回滚机制重建）。

### 中风险

1. SSE 长连接稳定性与工具事件流解析。
2. 截图/权限相关跨平台兼容性。
3. 与现有配置体系（`config.json`）的一致性读取与热更新策略。

### 低风险

1. 常规页面 UI 重建（配置页、列表页、表单页）。
2. 与后端 REST 对接（已有接口较完整）。

## 6. 工期与人力估算（方案 A）

前提：

- 目标平台先聚焦 Windows。
- 后端协议保持兼容。
- Live2D 允许分阶段上线。

估算（2~3 名工程师）：

1. P0 架构设计与 PoC：2~4 周
2. P1 核心 UI + 聊天流 + 配置页：4~6 周
3. P2 桌面能力（托盘/悬浮球/截图/开机自启）：4~6 周
4. P3 功能补齐（技能工坊/意识海/更新机制）：4~8 周
5. P4 稳定性与发布验证：2~4 周

总计：约 4~7 个月（含联调与稳定性修复）。

若包含 Live2D 原生深度复刻：再增加约 4~8 周。

## 7. 建议分阶段实施计划

### 阶段 0：冻结接口

1. 冻结前端所需 API 协议（聊天流、会话、配置、工具状态）。
2. 输出接口契约文档与示例流数据。

### 阶段 1：最小可用版本（MVP）

1. 登录/聊天/配置三大主流程打通。
2. 支持 SSE 聊天流与基础工具状态展示。
3. 保证可替代 Electron 的日常主流程。

### 阶段 2：桌面能力对齐

1. 完成托盘、快捷操作、开机自启、截图面板。
2. 复刻悬浮球基础能力（至少 ball/compact/full）。

### 阶段 3：高级能力与体验对齐

1. 意识海可视化、技能工坊、升级机制。
2. Live2D（若纳入本期）。

### 阶段 4：并行灰度与切换

1. Electron 与 Qt 双版本并行一段时间。
2. 依据崩溃率、响应时延、用户反馈决定正式切换。

## 8. 关键决策建议

1. 先确定“是否必须首期复刻 Live2D”，该项对周期影响最大。
2. 先做方案 A，不要在第一阶段扩大到后端重写。
3. 将“协议兼容测试”纳入持续集成，确保 Qt 前端长期可替换。
4. 为悬浮球、截图、更新机制单独设计回归测试场景。

## 9. 最终建议

如果目标是“提升桌面端稳定性/性能并逐步脱离 Electron”，Qt 改造是可行的。

最稳妥策略是：

1. 保留现有 Python 后端与接口契约。
2. 用 Qt 分阶段替换前端。
3. 在业务功能可用后，再评估是否有必要推进更深层 C++ 化。
