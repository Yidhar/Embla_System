# 13 安全盲区与加固基线（面向当前实现）


> 口径说明（archived）

文档状态：安全审计基线（As-Is 风险识别 + 强制加固）  
最后更新：2026-02-23  
适用范围：`apiserver/`、`system/`、`mcpserver/`、`agents/`、`core/`（`autonomous/` 为 archived 追溯路径）

## 1. 目标

将已识别的高风险盲区从“讨论项”升级为“可执行约束”，避免以下两类问题：

1. 设计文档与当前实现口径不一致，导致错误安全预期。
2. 仅靠提示词或软规则约束，缺少运行时强制控制。

## 3. 风险明细与强制控制

### 3.1 R1 Regex 黑名单绕过（Command Obfuscation）

攻击面：

1. 变量拼接：`x="r"; y="m"; $x$y -rf /`
2. 解码执行：`echo "..." | base64 -d | sh`
3. 跨解释器执行：`python -c ...`、`powershell -EncodedCommand ...`

问题本质：

1. 词法黑名单无法覆盖语义等价变体。
2. 解释器可作为“二次执行器”绕过 token 规则。

强制控制（必须并存）：

1. `run_cmd` 禁止直接执行自由字符串高风险命令，优先使用白名单 `argv` 工具。
2. 对解释器入口加硬门禁：拒绝 `python -c/-m`、`powershell -Command/-EncodedCommand` 等动态执行模式（仅允许受控子命令）。
3. 命令策略从“黑名单”升级为“能力白名单 + 参数 schema + 风险等级”。
4. 高风险动作（写仓库/部署/系统变更）必须经审批与审计链路。

验收标准：

1. 对不少于 20 组混淆 payload 的回归测试全部拒绝。
2. 日志中必须记录拒绝原因与命令指纹（不可仅返回通用错误）。

### 3.2 R2 `register_new_tool` 宿主劫持

攻击面：

1. 不受信插件代码在 MCP Host 主进程导入执行。
2. 插件可利用语言动态能力突破静态扫描。

问题本质：

1. “代码加载”即“代码执行”。
2. 主进程共享地址空间意味着一次突破可接管全局。

强制控制（必须并存）：

1. 禁止在主进程内直接加载不受信插件（No In-Process Untrusted Import）。
2. 插件运行必须隔离到独立进程/容器，使用最小权限与资源配额。
3. 插件注册采用签名校验与显式 allowlist，禁止匿名热加载。
4. 插件 IPC 仅允许结构化协议，不允许获得宿主内部对象引用。

验收标准：

1. 恶意插件样例无法影响宿主进程内存与控制流。
2. 插件崩溃或超时不会导致 MCP Host 退出。

### 3.3 R3 锁泄漏与物理层 fencing 失控

攻击面：

1. 全局锁持有者异常退出（如 `kill -9`）导致锁悬挂。
2. 旧 epoch 的物理子进程继续运行，和新 owner 产生冲突。

问题本质：

1. 仅靠逻辑状态机无法自动回收物理副作用。
2. 缺少“进程树级别”的 epoch 绑定与回收。

强制控制（必须并存）：

1. 全局互斥锁必须具备 TTL + 心跳续租 + fencing token。
2. 锁获取与释放必须幂等，系统启动时执行 orphan-lock 清道夫。
3. 所有可变更宿主状态的执行任务必须记录 `epoch` 并绑定 process group。
4. lease/fencing 失效时，强制 kill 对应旧 epoch 的整棵进程树。

验收标准：

1. 注入 `kill -9` 故障后，锁能在 TTL 内自动回收。
2. 主从切换后，旧 owner 的后台任务不会继续执行物理写操作。

### 3.4 R4 暴力截断导致结构化数据破损

攻击面：

1. 大型 JSON/CSV/XML 响应被字符串截断后变成无效数据。
2. LLM 将无效数据误判为命令失败，触发无效重试循环。

问题本质：

1. 结构化数据与纯文本日志不能使用同一截断策略。
2. 截断结果继续回注给模型会放大错误。

强制控制（必须并存）：

1. 结构化输出必须“保真传递”：落盘 artifact + 元数据引用，不做字符级切断。
2. 前端和模型默认消费摘要视图（schema/键统计/样本片段），按需拉取原文。
3. 纯文本输出允许截断，但必须声明 `truncated=true` 与 `total_chars/lines`。
4. 工具返回应区分 `raw_result_ref` 与 `display_preview` 两条通道。

验收标准：

1. `kubectl -o json`、`curl` 大响应在主链路可被稳定解析。
2. 截断状态不会诱发超过阈值的重复重试回路。

### 3.5 R5 自我进化评测毒化（Test Poisoning）

攻击面：

1. Agent 为“通过验证”直接修改测试或放宽断言。
2. 局部测试通过但能力整体退化。

问题本质：

1. “被测代码”和“测试裁判”同源可写，形成博弈漏洞。
2. 仅用单一测试集难以防过拟合。

强制控制（必须并存）：

1. 验证基线与业务变更解耦：关键测试目录默认只读，变更需额外审批。
2. 引入“裁判测试集”（golden suite），禁止由被测任务改写。
3. 验证采用双轨：工作区测试 + 干净检出重跑（clean checkout）。
4. 关键指标门槛（覆盖率下限、失败用例不允许净减少为 0 异常模式）。

验收标准：

1. “修改测试骗过验证”样例必须被识别并拒绝合并。
2. 回归基线漂移需要明确审计记录与人工确认。

### 3.6 R6 `sleep_and_watch` ReDoS + 日志轮转假死

攻击面：

1. 监听正则可被构造为灾难性回溯模式（如 `(a+)+$`），在长日志上打满 CPU。
2. 仅依赖 `tail -f` 监听旧 fd，logrotate 后不再收到新日志。
3. Agent 永久休眠，不再唤醒。

问题本质：

1. 正则匹配器缺少复杂度约束与超时保护。
2. 文件句柄语义与路径语义不一致（rotate 后 inode 已变化）。

强制控制（必须并存）：

1. 正则入口必须做复杂度门禁：长度上限、禁用高风险嵌套量词、编译失败即拒绝。
2. 匹配执行必须具备单次超时与总预算上限，超限进入降级或熔断。
3. 监听器必须实现 `tail -F` 语义（路径重开 + inode 变更检测）。
4. 轮转、截断、权限变化都要触发 watcher 状态迁移与重试。
5. 每个 watch 任务必须有心跳与最大休眠 TTL，超时自动告警。
6. 唤醒事件需具备幂等键，防止重复唤醒风暴。

验收标准：

1. 使用灾难性回溯样例时，watcher CPU 占用可被限流且不会拖垮宿主。
2. 人工触发 logrotate 后仍可按条件成功唤醒。
3. watcher 卡死可被监控并自动恢复。

### 3.7 R7 爆炸半径控制强依赖 ZFS/Btrfs

攻击面：

1. 在 ext4/xfs 公有云 ECS、标准 Docker 容器中无法执行 `zfs snapshot`。
2. 快照能力不可用时若静默降级，系统在高风险动作前失去恢复点。

问题本质：

1. 将“可选平台能力”当成“通用安全基座”。
2. 缺少多后端快照策略与显式降级协议。

强制控制（必须并存）：

1. 快照层必须实现后端能力探测与优先级链：ZFS/Btrfs -> LVM/Volume Snapshot -> 应用层归档快照。
2. 若仅能降级到应用层快照，必须提升风险等级并强制审批，不得静默放行。
3. 每次高风险执行必须记录 `snapshot_backend`、`restore_point`、`degrade_reason` 到审计日志。
4. 快照恢复路径必须在各后端独立做恢复演练（不只做创建成功检查）。

验收标准：

1. 在 ext4/xfs 与容器场景下，高风险动作前仍能生成可恢复的 `restore_point`。
2. 随机抽样回滚演练成功率达到设定阈值（例如 >= 99%）。

### 3.8 R8 GC 摘要压缩导致关键排障硬数据丢失

攻击面：

1. GC 将长上下文压缩为短摘要时丢弃 TraceID、Error Code、内存地址、长路径等硬证据。
2. 主模型唤醒后基于不完整信息反复重试，进入“幻觉排障”循环。

问题本质：

1. 将“可叙述摘要”误当“可取证事实”。
2. 缺少摘要通道与证据通道的分离设计。

强制控制（必须并存）：

1. GC 必须双通道产物：`forensic_artifact_ref`（原始证据）+ `narrative_summary`（摘要索引）。
2. 关键字段必须强制保留并可检索：`trace_id`、`error_code`、`stack_frame`、`hex_address`、`full_path`。
3. 摘要模型不得改写原始证据，只允许生成引用与索引，引用需可回跳到原文偏移。
4. 唤醒注入时优先注入“证据索引卡片”，并保留按需拉取原文接口。

验收标准：

1. 压测场景下，主模型可通过 `forensic_artifact_ref` 复原关键排障链路。
2. 关键字段召回率满足阈值（例如 >= 99%）。

### 3.9 R9 `raw_result_ref` 读后即盲

攻击面：

1. 工具仅返回 `display_preview + raw_result_ref`，但无读取工具或读取协议。
2. 根因字段藏在大对象中后段，模型无法二次取证。

问题本质：

1. “可引用”不等于“可访问”。
2. 缺少引用解析层会把可用数据变成不可达数据。

强制控制（必须并存）：

1. 返回 `raw_result_ref` 时必须同时提供 `artifact_reader` 能力（`jsonpath/line_range/grep`）。
2. `raw_result_ref` 必须携带 `content_type/size/created_at/ttl/fetch_hints` 元数据。
3. 模型侧必须支持“预览不足 -> 二次读取”的标准回路，不得直接盲重试命令。

验收标准：

1. 对超大 JSON 用例，模型可通过二次读取定位中段 `error_code/uuid` 并完成排障。
2. “有 ref 但无读取路径”的工具结果在契约检查阶段被拒绝。

### 3.10 R10 `file_ast` 巨型单体文件认知迷雾（Monolith OOM）

攻击面：

1. 大文件（如 30,000 行）编辑前若全量回读，会触发上下文爆炸与 GC 抖动。
2. 简单截断会损失全局符号拓扑，定位函数签名失真。

问题本质：

1. 编辑型任务把“读取全量正文”当默认路径。
2. 缺少分层代码视图（索引/骨架/局部正文）。

强制控制（必须并存）：

1. 超大文件默认走 `file_ast_skeleton`：仅返回 imports/symbols/function_ranges。
2. 正文读取必须按目标范围分块（`readChunkByRange`），禁止默认全量读取。
3. 分层读取结果需带 `symbol_checksum`，用于后续编辑一致性校验。

验收标准：

1. 30k 行单体文件场景下，编辑链路不依赖全量正文回读即可完成定位与修改。
2. 上下文 token 峰值控制在设定阈值内（例如 < 10k 动态窗口）。

### 3.11 R11 `file_ast` 并发活锁（Livelock Storm）

攻击面：

1. 多 Agent 同时改同一文件，hash 频繁失配。
2. 模型反复“重读 -> 重写 -> 再冲突”，预算快速耗尽。

问题本质：

1. 冲突后处理路径只有“硬失败 + 重试”。
2. 缺少冲突票据、退避、语义重基。

强制控制（必须并存）：

1. hash 冲突后优先走 `semantic_rebase` 快速路径。
2. 冲突重试必须附带 `conflict_ticket + backoff_ms`，执行指数退避和抖动。
3. 达到重试上限后升级到 Router 协调或人工仲裁，禁止继续盲写。

验收标准：

1. 并发编辑压测下，冲突场景不出现无限重试环。
2. 预算消耗曲线在冲突高峰期仍保持可控。

### 3.12 R12 Sub-Agent 并行盲写（缺少契约谈判屏障）

攻击面：

1. FE/BE 子代理并行产出补丁，但未先冻结 API 契约。
2. 合并后字段错配，编译与联调循环失败。

问题本质：

1. 并行执行先于契约协商。
2. 缺少跨子代理一致性前置门禁。

强制控制（必须并存）：

1. 引入 `Contract Gate`：并行分析前必须生成并确认 `contract_id + checksum`。
2. 未达成契约时 fail-fast，禁止进入脚手架阶段。
3. 生成补丁与验证步骤必须绑定同一 `contract_id`。

验收标准：

1. FE/BE 双侧字段错配在生成阶段即被拦截，不进入编译环。
2. 合并后首轮联调成功率显著提升并可度量。

### 3.13 R13 脚手架跨文件非原子提交（Non-atomic Scaffolding）

攻击面：

1. 多文件补丁中途失败，工作区进入半写损坏态。
2. 后续 Agent 难以恢复一致状态，修复复杂度指数上升。

问题本质：

1. 批量写入缺少事务语义。
2. 失败后无系统级自动回滚。

强制控制（必须并存）：

1. 多文件 scaffold 写入必须 `begin -> apply_all -> verify -> commit/rollback`。
2. 任一文件失败立即全量回滚，并返回 `clean_state=true`。
3. 提交前必须执行最小编译/测试门禁，禁止把破碎状态暴露给下一轮。

验收标准：

1. 注入第 N 个文件失败时，工作区可自动回到操作前一致状态。
2. 回滚后可直接进行下一次重试，不需人工清理残局。

### 3.14 R14 KillSwitch 缺少 OOB 导致云主机失联

攻击面：

1. 一刀切 `OUTPUT DROP` 切断 SSH 回包、堡垒机链路和云管通道。
2. 故障时无法远程执行 `disarm/recover`，形成运维孤岛。

问题本质：

1. 网络熔断策略未区分业务出口与管理平面。
2. 缺少带外（OOB）保活与健康探测。

强制控制（必须并存）：

1. 熔断时执行“默认拒绝 + OOB allowlist”策略，而非全量 DROP。
2. 维护 OOB 管理平面白名单（CIDR/域名/端口）与健康探测。
3. 每次熔断前后都要校验 OOB 可达性，失败时进入降级策略并升级告警。

验收标准：

1. 模拟熔断时，堡垒机/SSM/云控制台通道保持可用。
2. 管理员可在 OOB 通道内完成 `disarm/recover`。

### 3.15 R15 Fencing 强杀被 Double-Fork 幽灵逃逸

攻击面：

1. `docker run -d`、`nohup`、`setsid` 派生后台任务脱离原 PGID。
2. 旧 epoch 后台进程持续写入或占端口，击穿单活保障。

问题本质：

1. 回收机制仅覆盖进程组，不覆盖容器/cgroup/命名空间。
2. 执行任务缺少统一 lineage 标识。

强制控制（必须并存）：

1. 全局动作必须绑定 `job_root_id + cgroup/container_id`。
2. fencing 失效时按 lineage 做递归清理（PGID + cgroup + runtime API）。
3. 对 detached 任务设置续租心跳，失联即回收。

验收标准：

1. 双重派生与 detached 容器场景下，旧 epoch 进程可被稳定回收。
2. 主备切换后不存在旧任务持续占端口/写文件现象。

### 3.16 R16 Artifact Store 磁盘 DoS

攻击面：

1. 超长输出和原始证据持续落盘导致磁盘快速耗尽。
2. 关键组件（SQLite/EventBus）因 `ENOSPC` 受损，触发系统级雪崩。

问题本质：

1. Artifact 写入只有“入站”没有“淘汰与限流”。
2. 缺少高水位熔断与优先级清理。

强制控制（必须并存）：

1. 引入 `global/session/tenant` 三层配额 + TTL + LRU 回收。
2. 高水位触发写入背压：拒绝低优先级大对象并强制告警。
3. 关键存储路径隔离（Artifact 与 SQLite/EventBus 分区），避免连带损坏。
4. 清理策略必须审计化并支持恢复演练。

验收标准：

1. 72h 无人值守压测中，磁盘使用维持在安全阈值内。
2. 压测触发高水位时，系统可降级运行且核心数据库不损坏。

## 4. 分阶段落地顺序

### Phase P0（先止血）

1. 禁止不受控自由命令执行路径。
2. 修复结构化输出破损链路（artifact + preview 双通道）。
3. 明确并拦截测试毒化高风险改动。
4. 为 `sleep_and_watch` 增加正则复杂度门禁与匹配超时。
5. 为 `raw_result_ref` 补齐 `artifact_reader` 二次读取链路。
6. 为 KillSwitch 增加 OOB 白名单保活策略。

### Phase P1（补治理）

1. 全局锁 TTL/fencing + orphan 清理。
2. 物理层进程组回收与旧 epoch 强杀。
3. 插件执行隔离与签名注册。
4. 快照后端能力探测与降级审批链路。
5. `file_ast` 分层读取 + 语义重基 + 冲突退避治理。
6. Sub-Agent `Contract Gate` 与 Scaffold 事务提交能力。

### Phase P2（长期自治能力）

1. `sleep_and_watch` 生产级轮转容错与 ReDoS 压力基线。
2. GC 证据通道质量指标（关键字段召回率）纳入回归。
3. Artifact Store 生命周期治理（配额/TTL/高水位熔断）稳定化。
4. 完整审计看板与自动化安全回归基线。
